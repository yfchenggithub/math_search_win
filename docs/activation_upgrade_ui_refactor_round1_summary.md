### 2. 当前数据来源识别
- 当前项目尚未在 UI 层接入完整 license service 数据流。
- 本轮页面展示数据主要来自：
  - `UiConstants`（版本与模式文本）
  - 现有占位字段（授权类型、有效期、设备码、升级说明）
- 结论：本轮可做结构与表现升级，但不应触碰授权校验、设备码生成、激活策略、升级策略等底层逻辑。

- 激活页已是独立页面类，注册于 `QStackedWidget`，无需新增页面路由体系。
- 首页已有“激活与升级”入口，保持不变。

### 4. 最小侵入改造策略
- 保留页面类与路由，不改导航系统。
- 重构 `ActivationPage` 的布局与分区组织。
- 通过 `objectName` + `property` 在 `app.qss` 增量补充样式。
- `MainWindow` 仅做轻微刷新增强：切到激活页时触发 `reloadData()`。

## 第三部分：设计说明（重点）

### 1. 为什么按“状态确认 + 激活操作 + 升级说明”分区
- 这是用户在授权入口最自然的决策顺序：
  1. 先确认“我当前是什么状态”
  2. 再执行“我下一步怎么做”
  3. 最后了解“我后续可升级到哪里”
- 避免把关键流程压缩成“单行输入框 + 按钮”的表单感，提升入口页的正式度与可理解性。

### 2. 为什么状态区必须前置
- 授权状态是后续操作前提，不先看到状态会导致操作不确定。
- 前置状态区可以降低误操作成本（例如已激活用户不必重复尝试激活）。
- 同时支持后续扩展：已激活、未激活、未知、异常四种状态都能自然承载。

### 3. 为什么设备码要独立承托
- 设备码是授权识别关键字段，不能与普通信息混排成“弱字段”。
- 采用独立分组 + 只读输入框样式，使其在视觉上具备“重要可复制识别码”特征。
- 这为后续“复制设备码”功能提供天然挂载点，无需二次大改。

### 4. 为什么升级说明不能做成营销块
- 当前未接入完整商业化升级能力，做价格卡/促销卡会造成“伪功能”与信任损耗。
- 本轮升级说明定位为“产品路径说明”，保持克制，保证信息可信与界面一致性。
- 升级按钮明确设为次主操作，避免与“立即激活”主路径竞争注意力。

### 5. 哪些点提升了“可信赖授权入口”的感受
- 页面头部层级明确（标题/副标题/元信息）。
- 分组语义清晰（状态、操作、升级职责分离）。
- 授权状态采用可扩展视觉态而非普通字段值。
- 设备码与激活输入区具有正式表单承托感。
- CTA 层级清楚：激活主按钮 > 升级次按钮。
- 文案克制，不堆砌营销或伪功能。

### 6. 为后续真实授权流程预留了哪些扩展点
- 授权状态态机骨架已在 UI 层具备承载结构。
- 设备码区可直接加“复制”与“复制反馈”。
- 激活输入区可接“粘贴”、“激活结果反馈条”、“错误码提示”。
- 授权状态区可补充授权类型、到期时间、激活时间、通道信息。
- 升级区可接真实升级方案详情，不需要改整体布局。

## 第四部分：后续可选优化（重点，仅列最值得做的 5 项）

### 1. 接入复制设备码
- 在设备码区增加“复制”按钮与“已复制”短反馈。
- 价值：减少手工抄写错误，提升支持流程效率。

### 2. 接入激活结果反馈条
- 激活后展示成功/失败/异常反馈（含错误原因）。
- 价值：把“点击后无反馈”的不确定性转为可解释状态。

### 3. 接入真实授权字段
- 补齐授权类型、有效期、激活时间等字段。
- 价值：让“授权状态区”从骨架升级为可运营、可排障信息面板。

### 4. 接入升级方案详情
- “查看升级方案”对接真实方案页或弹层（版本差异、适用人群、变更范围）。
- 价值：让升级路径具备可执行闭环，不停留在占位入口。

### 5. 接入轻量授权帮助区
- 增加折叠式“常见问题/离线激活说明”。
- 价值：降低支持压力，同时不破坏页面克制感与主路径清晰度。
